# Zava Learning — Weekly Audit Report Template (`zava-audit-report`)

The canonical structure for the **proactive weekly governance audits** (NSG / network security,
RBAC / least-privilege, and Cloud cost). These are NOT incident reports — there is no PagerDuty
incident, no 5-Whys, no before/after recovery. Each audit is a posture review of the Zava Learning
resource group that produces one branded, downloadable PowerPoint deck for platform owners.

Audit skills retrieve this with `SearchMemory` (memory name `zava-audit-report`) and fill each
section from their findings. Apply the `zava-brand` standard for all styling. Omit a section only
when it has no content; never reorder.

## Voice & tone
- Governance/assurance voice: factual, calm, executive-ready. No blame.
- Lead with posture (how healthy is this area) and the few things that matter most.
- Quantify everything: counts of findings by severity, % of identities over-privileged, $ spend and
  week-over-week delta, number of open management ports, etc.
- Every finding is actionable: each row names the resource, the issue, the risk, and the fix.
- Read-only assurance: the audit recommends; it never changes anything. State this explicitly.

## Severity model (use consistently, impact-based)
`SEV1 Critical` (Critical red) · `SEV2 Major` (Warning amber) · `SEV3 Minor` (Slate). Examples:
- NSG: any-source inbound to a management port (22/3389) or `Allow *:* *` = SEV1; broad CIDR to a
  data port = SEV2; legacy/unused or shadowed rule = SEV3.
- RBAC: Owner/Contributor granted directly to a user (not a group), or to a guest/stale principal =
  SEV1; over-broad scope (subscription where RG suffices) = SEV2; direct assignment that should be
  group-governed = SEV3.
- Cost: a resource trending sharply up week-over-week or an idle/oversized resource = SEV2; smaller
  optimisation opportunities = SEV3.

## Canonical audit deck order (16:9, one idea per slide)

1. **Title** — `<Audit Type> Audit` + ZAVA wordmark + resource group + UTC date + "Weekly automated
   governance review". Indigo title bar, white title.
2. **Posture summary** — 3–5 executive bullets: overall posture (Healthy / Needs attention /
   At risk), how many items reviewed, how many findings by severity, the single most important
   action. A VP reads only this slide.
3. **Findings at a glance** — ONE chart that shows the shape of the result: a bar/column chart of
   findings by severity (SEV1/SEV2/SEV3) using brand colors (Critical red / Warning amber / Slate),
   or for cost a bar chart of top cost drivers. Use `PlotBarChart` / `PlotPieChart`.
4. **Findings detail** — the core table, sorted by severity (SEV1 first). Columns per audit type:
   - **NSG:** `NSG · Rule · Direction · Source → Dest:Port · Finding · Severity · Recommended fix`.
   - **RBAC:** `Identity (type) · Role · Scope · Assignment (direct/group) · Finding · Severity ·
     Recommended role`.
   - **Cost:** `Resource (type) · This week $ · Last week $ · Δ% · Finding · Severity · Recommendation`.
   Zebra-stripe rows in Mist; severity cell colored. Split across slides if more than ~8 rows.
5. **Trend / context** (optional) — week-over-week movement where available (cost trend line; new vs.
   resolved findings vs. last week for NSG/RBAC if a prior run is known). Use a dashed Slate baseline.
6. **Recommendations & next steps** — prioritized action table: `Action · Owner · Priority ·
   Effort · Target date`. Group by severity; SEV1 actions first. Read-only: these are recommendations
   for a human to apply, never auto-changes.
7. **Closing** — overall posture statement, scope/limitations (what was and wasn't covered), the
   read-only assurance note, contacts, and the confidentiality footer.

## Quality bar (every deck must pass)
- Title is the audit type only (e.g. "Network Security Group Audit") — no alarming root-cause
  language; this is a posture review, not an incident.
- Every finding row has a concrete recommended fix and an impact-based severity.
- The "Findings at a glance" visual is really rendered, brand-colored, and labeled.
- Counts are consistent across the summary, the chart, and the detail table.
- No secrets, keys, connection strings, or learner PII — apply the `zava-redaction` scrub
  (`SearchMemory("zava-redaction")` → `redact()`) to every slide's text before writing the file.
- Confidentiality footer (`Confidential — Internal Use Only · Generated by the Zava SRE Agent`) and
  the header line (`ZAVA Learning · Site Reliability · <Audit Type> Audit · <UTC date>`) present.
- The deck is emitted as a real downloadable `.pptx` attachment (see `zava-audit-report` skill) with
  a working download link surfaced to the operator.
